Vue d'ensemble — Dernière mise à jour : 24 avril 2026

Digital Humans est une plateforme SaaS B2B qui automatise le développement Salesforce grâce à 11 agents IA spécialisés. Le système génère des spécifications complètes (SDS) puis du code Salesforce déployable (BUILD) via un pipeline multi-phases orchestré avec validation humaine à chaque étape clé.

Pipeline SDS (5 phases)

Sophie → Olivia → Emma → Marcus ↔ Emma → Experts → SDS

193 fichiers Python 40 fichiers React

Pipeline BUILD (6 phases, 4 agents)

Raj → Diego → Zara → Raj → Raj → Aisha + Jordan (deploy) + Elena (review)

Sonnet 4.5 pour tous Opus pour architecture

Worker async (ARQ + Redis)

Toutes les exécutions SDS/BUILD tournent en background via ARQ workers avec suivi temps réel via polling/SSE

3 tâches : SDS, Resume, Build

HITL — Human in the Loop

5 gates de validation (BR, Architecture, Experts, SDS, BUILD) + chat contextuel multi-agent (9 agents)

State Machine 27 états

Phase SDS — Spécification

Le pipeline SDS transforme un document métier en spécification Salesforce complète via 5 phases avec validation humaine (HITL).

Révision Architecture (fix_gaps)

Quand Emma détecte un coverage < 85%, elle génère des gaps (what_is_missing + fix_instruction). Marcus reçoit ces gaps en mode fix_gaps (incrémental). Boucle auto-revision max 3 itérations.

Phase BUILD — Génération de code

Le BUILD transforme les specs SDS en code Salesforce déployable. Chaque agent BUILD utilise Sonnet 4.5 et reçoit les specs détaillées de Marcus comme contexte. Raj (Admin) intervient sur 3 phases, Jordan orchestre le déploiement de chaque phase via JordanDeployService, et Elena review chaque output avant deploy.

Change Requests (CR)

Lifecycle complet : draft → submitted → analyzed → approved/rejected → processing → completed. Sophie détecte les demandes de CR dans le chat, crée le CR, analyse l'impact (agents à rerun + coût estimé), et l'utilisateur approuve/rejette.

Problèmes architecturaux (P0-P11)

12 problèmes identifiés (audit initial + P10-P11 ajoutés post-E2E). 10 fixés via Sprints 0-3 + Session 3 Refonte (18 avril), 2 partiels restants.

✅ Fixés (10)

P0 Routes async avec SQLAlchemy sync bloquaient l'event loop (S0, A-7 / A-8 (Session 3))
P1 Split brain — pm.py v1 et pm_orchestrator.py v2 cohabitaient (S0)
P2 52 chemins absolus hardcodés (S0, D-1a (Session 3))
P3 Agents lancés via subprocess.run() — overhead 3-5s/appel (pre-E2E #145)
P5 Logs fragmentés — import logging basique partout (D-2 (Session 3))
P6 13 modèles LLM hardcodés dans le code (Track A, C-5 (Session 3))
P7 Transactions non-atomiques — 24 db.commit() éparpillés (A-9 (Session 3))
P8 Rotation de secrets absente (SECRET_KEY, ANTHROPIC_API_KEY) (S3, D-3 (Session 3))
P9 safe_content() tronquait 93-96% du contenu (Track A)
P11 RAG health silencieux — outages non détectés (A-10 (Session 3))
