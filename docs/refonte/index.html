Vue d'ensemble — Dernière mise à jour : 24 avril 2026

Digital Humans est une plateforme SaaS B2B qui automatise le développement Salesforce grâce à 11 agents IA spécialisés. Le système génère des spécifications complètes (SDS) puis du code Salesforce déployable (BUILD) via un pipeline multi-phases orchestré avec validation humaine à chaque étape clé.

Pipeline SDS (5 phases)

Sophie → Olivia → Emma → Marcus ↔ Emma → Experts → SDS

144 fichiers Python 40 composants React

Pipeline BUILD (6 phases, 4 agents)

Raj → Diego → Zara → Raj → Raj → Aisha + Jordan (deploy) + Elena (review)

Sonnet 4.5 pour tous Opus pour architecture

Worker async (ARQ + Redis)

Toutes les exécutions SDS/BUILD tournent en background via ARQ workers avec suivi temps réel via polling/SSE

3 tâches : SDS, Resume, Build

Découverte transversale — le templating corrige le pipeline

En cherchant le pixel-near sur la référence, on a découvert que l'ancien pipeline phase 5 Emma produisait des bugs systématiques. Notre rendu DB-driven les corrige par construction :

  • ~30 corrections d'API names Salesforce : Vehiclec, Sinistrec, Maintenance_Activityc, Alert_Configurationmdt → Vehicle__c, Sinistre__c, etc. Bug racine : parser markdown qui interprétait les __ comme italique dans les textes descriptifs.
  • 4 sections avec colonnes vides corrigées : OWD (Internal+External), Reports Purpose (Object/Filters/Grouping), Flows Trigger (event+condition), UI Components Purpose.
  • 5 sous-sections manquantes ajoutées en 6.8 et 6.10 : Apex Triggers (3), Scheduled Jobs (9), Platform Events (1), LWC Components (13), Quick Actions (6).
  • 1 phrase tronquée restaurée dans 6.12 Tech Considerations.

Pièges connus

  • JSONB ordre des clés : PostgreSQL ne préserve pas l'ordre d'insertion. Pour tout rendu k-v vertical, définir ordered_keys explicite dans le partial.
  • Jinja2 dict.values piège : la résolution attribut Python prend field.values pour la méthode built-in avant la clé 'values'. Utiliser field['values'] et 'values' in field. Idem pour keys et items.
  • Mermaid déjà chargé dans le shell : ne PAS rajouter de <script> dans les partials — l'init existe avec la palette Studio.
  • Référence figée : _reference_logifleet_146.html ne doit JAMAIS être régénéré. C'est notre baseline diff. Pour une nouvelle référence, créer un fichier différent.
  • StrictUndefined : si une clé est légitimement optionnelle, l'envelopper dans {% if key %} avant accès.
  • Le HTML de référence a des bugs : ne PAS les copier. Le diff métrique augmente avec les corrections, c'est normal.
  • Connection PostgreSQL hardcodée dans collect_sds.py. À déporter dans .env à la bascule prod.
  • Filtre humanize à étendre : si une nouvelle clé snake_case ambiguë apparaît, ajouter un mapping dans SPECIAL de tools/build_sds.py.

Logging structuré (session 3 — P5)

Depuis le 18 avril, tous les services utilisent un ExecutionContextMiddleware qui injecte automatiquement execution_id et agent_id dans les LogRecord. Un grep par execution_id=1234 donne toute la trace d'une exécution à travers les processes (backend + worker).

# Suivre une exécution précise en temps réel
journalctl -u digital-humans-backend -u digital-humans-worker -f --no-pager | grep 'execution_id=1234'

# Voir toutes les actions d'un agent sur la dernière heure
journalctl -u digital-humans-worker --since "1 hour ago" | grep 'agent_id=marcus'

Budget & guardrails

Deux services protègent la plateforme contre les runaway coûts et les boucles infinies d'agent : BudgetService (limites financières hiérarchiques) et CircuitBreaker (limites comportementales). Les deux vivent dans backend/app/services/budget_service.py.

BudgetService — limites financières

Appelé avant chaque appel LLM (check_budget) et après chaque réponse (record_cost). Trois niveaux hiérarchiques : si un niveau est dépassé, BudgetExceededError est levée et l'exécution est mise en FAILED.

NiveauLimitePortéeReset
Execution$50 USDUne exécution SDS ou BUILDPar execution_id
Project$200 USDSomme de toutes les exécutions d'un projetPar project_id
Monthly$500 USDTous projets confondusDébut de mois calendaire

Les limites sont actuellement hardcodées (DEFAULT_EXECUTION_LIMIT_USD, etc.). Backlog : rendre paramétrable par projet via projects.budget_limits (table existante, pas encore consommée).

Pricing — source unique YAML

Depuis la session 3 (Agent C), BudgetService lit les tarifs depuis llm_routing.yaml. Les clés sont indexées à la fois par provider/alias (ex: anthropic/claude-opus) et par model_id résolu (ex: claude-opus-4-6) pour que les appelants puissent passer l'une ou l'autre forme. Voir la table de routage LLM pour les tarifs live.

CircuitBreaker — limites comportementales

Prévient les boucles infinies agent ↔ agent (typiquement le loop Marcus ↔ Emma en cas de coverage qui ne monte jamais au-dessus du seuil).

ContrainteSeuilComportement au dépassement
MAX_AGENT_RETRIES3 par agent et par exécutioncheck_agent_retry retourne False, l'exécution marque l'agent en failed
MAX_REVISION_LOOPS2 Emma ↔ Marcus maxAprès 2 allers-retours, Emma écrit le SDS avec la meilleure version Marcus disponible
MAX_TOTAL_LLM_CALLS80 par exécution (~43 normal pour SDS)Exécution mise en FAILED avec failure_reason="circuit_breaker_total_calls"

Observabilité coûts

Chaque appel LLM est tracé dans la table llm_interactions (18 colonnes) avec model, input_tokens, output_tokens, cost_usd. Les coûts sont aussi agrégés :

  • executions.cost_usd — rolling sum par exécution (refresh sur record_cost)
  • executions.budget_used — fraction [0..1] du DEFAULT_EXECUTION_LIMIT_USD
  • execution_agents.tokens_used / .cost_usd — ventilation par agent

Voir la section Observability pour les requêtes SQL de monitoring coûts.
