Modèle de crédits

Implémenté dès la V1 (pas reporté). Source de vérité : tables credit_balances, credit_transactions, model_pricing, tier_config (cf. Base de données). Service centralisé : app/services/credit_service.py.

Une unité de crédit = 1 000 tokens d'input Sonnet équivalent. Conversion par modèle :

ModèleInput (cr / 1K tok)Output (cr / 1K tok)Tiers autorisésOpt-in
claude-haiku-4-50.3001.500free, pro, team—
claude-sonnet-4-61.0005.000pro, team—
claude-opus-4-75.00025.000team✅ requis

Allotements mensuels par tier (table tier_config) : Free 0 crédits inclus + cap journalier 300, Pro 2 000/mois, Team 100 000/mois. Reset mensuel par cron.

Offre retenue : Stripe Standard ("I'll do it"), pas Managed Payments. Tarifs négociés à l'ouverture du compte (29 avril 2026) :

Hook dans le LLM router

Avant chaque appel LLM, le router appelle credit_service.preflight_check(user_id, model, max_tokens) qui :

  1. Vérifie que le tier de l'utilisateur autorise le modèle (cf. model_pricing.allowed_tiers)
  2. Vérifie que le solde de crédits couvre l'estimation maximale (input + output)
  3. Lève ModelNotAllowedError ou InsufficientCreditsError avant tout appel API

Après l'appel, credit_service.charge() débite le solde réel (basé sur les tokens effectivement consommés) et inscrit une ligne dans credit_transactions avec lien vers execution_id et project_id.
