Procédure de rotation — ANTHROPIC_API_KEY

Livrée en session 3 (Agent D, 18 avr). Procédure manuelle, ~3 minutes, zero downtime sauf fenêtre de 5s pendant le restart.

# 1. Générer la nouvelle clé dans la console Anthropic
# 2. Éditer backend/.env, remplacer ANTHROPIC_API_KEY=...
nano /root/workspace/digital-humans-production/backend/.env

# 3. Restart backend + worker (la clé est lue au démarrage)
sudo systemctl restart digital-humans-backend digital-humans-worker

# 4. Smoke test
curl -s localhost:8002/api/health | jq .llm_provider_reachable
# Doit retourner true

# 5. Révoquer l'ancienne clé dans la console Anthropic

Pièges connus

  • P12 — deux fichiers .env OpenAI. backend/.env (consommé par l'app) et /opt/digital-humans/rag/.env (consommé par les scripts d'ingestion RAG). Rotation de OPENAI_API_KEY = modifier les 2. À consolider via dotenv centralisé (backlog).
  • Pas de rotation automatique. Toutes les rotations sont manuelles. Pour du production-grade, migrer vers un vault (Vault / AWS SM / Doppler) — hors scope à date.
  • JWT expire_minutes court. 30 min : OK pour un produit dev-facing, à remonter à quelques heures avec refresh tokens si on ouvre à des utilisateurs non-techniques.
  • Audit trail. La table audit_logs trace les actions utilisateur (22 cols). Les actions secrets (rotations, changements) ne sont pas tracées — à ajouter.

Hardening — backlog

  • Refresh tokens JWT (séparation access/refresh)
  • 2FA pour les comptes admin
  • Chiffrement at-rest des project_credentials (actuellement clair en DB)
  • Rate limiting sur /auth/login (brute force)
  • CSP strict dans les headers Nginx
  • Rotation auto SECRET_KEY + blacklist des tokens émis avec l'ancienne clé (impossible sans refresh tokens)
