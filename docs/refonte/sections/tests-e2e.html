Tests E2E

La campagne E2E est l'outil principal de validation produit. Chaque test numéroté (E2E #N) exécute un pipeline SDS complet sur un projet test réel (LogiFleet, flotte de véhicules B2B) et trace l'état dans docs/e2e-tests/E2E_N_STATE.json. Campagne en cours depuis février 2026, cadence mensuelle.

Méthodologie

ÉtapeAction
1. FreezeSnapshot git (tag pre-e2e-N), worker et backend en état connu, DB reset sur le projet test
2. RunLancement SDS Phase 1→5 via API. Monitoring live via journalctl -u digital-humans-backend -f.
3. State captureEn fin de run, sérialisation de l'execution row + deliverables + llm_interactions dans E2E_N_STATE.json
4. Post-mortemAnalyse des écarts attendus vs observés. Bugs catalogués dans E2E_N_REPORT.md s'il y en a.
5. HotfixesFix des bugs bloquants avant le E2E suivant. Fix non bloquants backlogués.

Enseignements

Les campagnes E2E ont révélé plus de bugs que les tests unitaires et d'intégration réunis. Cause probable : le pipeline SDS est intrinsèquement un système qui n'expose sa correction qu'à l'échelle du run complet. Les composants pris individuellement semblent tous OK ; ce sont les couplages (Emma scoring ↔ Marcus output format, safe_content ↔ taille des sections, record_cost ↔ propagation) qui cassent.

Backlog :

  • Automatiser la capture du state file (actuellement semi-manuel)
  • Benchmarks de référence (tokens, coût, durée) pour détecter les régressions de perf
  • Run quotidien automatique (cron + notification Slack si régression)
  • Diversifier les projets tests (pas seulement LogiFleet — ajouter un cas finance, un cas santé)
