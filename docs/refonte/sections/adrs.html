Architecture Decision Records

Les ADR consignent les décisions structurantes : contexte, options évaluées, décision retenue, conséquences. Les fichiers complets vivent dans docs/ADR-NNN-slug.md dans le repo. Ci-dessous le sommaire.

ADR-001 — LLM Strategy

MétaValeur
StatutAccepted
Date18 avril 2026 (session 3)
AuteurAgent C — consolidé par Agent D
Source de véritébackend/config/llm_routing.yaml
Fichierdocs/ADR-001-llm-strategy.md

Problème. Avant session 3, la couche LLM souffrait de : (1) tier sprawl — 5 tiers nominaux qui mappaient tous vers 2 classes de modèles dans les faits, (2) model IDs hardcodés dans 3 services différents, (3) pricing drift de 4× entre budget_service.MODEL_PRICING et la console Anthropic (mesuré en mars 2026), (4) pas de chemin on-premise pour les clients airgap.

Décision. Adopter 2 tiers logiques × 3 profiles de déploiement, tous exprimés dans un seul YAML :

  • Tiers : orchestrator (Sophie, Olivia, Marcus, Emma — raisonnement) · worker (Diego, Zara, Raj, Elena, Jordan, Aisha, Lucas — exécution)
  • Profiles : cloud (Opus/Sonnet), on-premise (Mixtral/Mistral via Ollama, zero cloud fallback), freemium (Sonnet/GPT-4o-mini avec cap mensuel)
  • Fallback chain par profile — jamais de cross-profile
  • Continuation auto sur stop_reason=max_tokens (porté de V1 CRIT-02)

Voir la section LLM — Multi-profile & routing pour les détails d'implémentation.

ADR-002 — Agents Registry

MétaValeur
StatutAccepted
Date18 avril 2026 (session 3)
AuteurAgent B — consolidé par Agent D
Source de véritébackend/config/agents_registry.yaml
Accessorbackend/app/services/agents_registry.py
Fichierdocs/ADR-002-agents-registry.md

Problème. Les métadonnées des agents vivaient dans 6 endroits découplés : AGENT_CONFIG, AGENT_COLLECTIONS, CATEGORY_AGENT_MAP, agent_artifact_needs, AGENT_CHAT_PROFILES, AGENT_COSTS. Symptômes : renommer Admin → Salesforce Admin nécessitait 6 PRs ; ajouter Lucas (11ème agent) a introduit des incohérences survenues à 3 code reviews (Lucas était dans AGENT_CONFIG mais absent de AGENT_COLLECTIONS — ses appels RAG tombaient silencieusement sur la collection par défaut).

Décision. Un seul YAML (agents_registry.yaml) avec une entrée par agent portant toutes les propriétés : name, role / role_fr, agent_type, tier, rag_collections, aliases, cost_estimate_usd, color, etc. Un accessor unique (app.services.agents_registry) — aucun autre service ne lit le YAML directement. Alias résolus : pm → sophie, architect → marcus, etc.

Voir la section Les 11 agents pour la table live.

Convention — proposer un ADR

Template : docs/ADR-NNN-slug.md, structure en 5 sections fixes (Status · Context · Decision · Consequences · Supersedes). Numérotation incrémentale, jamais de renumérotation. Un ADR en statut Superseded reste dans le repo — il fait partie de l'historique des décisions.
