Observability

Comment comprendre ce que fait la plateforme en prod : logs, métriques, audit trail, et tables de tracking. L'approche est délibérément low-tech (journalctl + PostgreSQL) plutôt que stack observability complète — ça tient tant que les incidents sont rares et investigués manuellement.

Logging structuré (session 3 — P5)

Depuis le 18 avril, tous les services utilisent un ExecutionContextMiddleware qui injecte automatiquement execution_id et agent_id dans les LogRecord. Un grep par execution_id=1234 donne toute la trace d'une exécution à travers les processes (backend + worker).

# Suivre une exécution précise en temps réel
journalctl -u digital-humans-backend -u digital-humans-worker -f --no-pager | grep 'execution_id=1234'

# Voir toutes les actions d'un agent sur la dernière heure
journalctl -u digital-humans-worker --since "1 hour ago" | grep 'agent_id=marcus'

Tables de tracking en DB

TableCe qu'on y trouveRequête utile
llm_interactions Chaque appel LLM (model, tokens in/out, cost, duration, prompt hash) SELECT agent_id, COUNT(*), SUM(cost_usd) FROM llm_interactions WHERE execution_id=1234 GROUP BY agent_id;
audit_logs Actions utilisateur : login, create project, launch execution, validate gate SELECT * FROM audit_logs WHERE user_id=5 ORDER BY created_at DESC LIMIT 20;
executions État de chaque run SDS/BUILD (status, state, cost_usd, budget_used, error_message) SELECT id, status, state, cost_usd, started_at FROM executions WHERE status='failed' ORDER BY started_at DESC;
execution_agents Ventilation par agent : tokens, cost, progress, iterations, status SELECT agent_id, tokens_used, cost_usd, iterations FROM execution_agents WHERE execution_id=1234;
validation_gates Décisions HITL (approve/reject/revise) avec feedback texte SELECT gate_type, decision, LENGTH(feedback) FROM validation_gates WHERE execution_id=1234;

Roadmap — Admin Hub

Prévu pour remplacer le bricolage journalctl + psql par une UI unifiée. Accessible sur digital-humans.fr/admin. Objectifs :

  • Statut UP/DOWN de tous les services (backend, worker, frontend, Postgres, Redis, Ollama, Nginx, Ghost, N8N, Open WebUI)
  • Boutons start/stop/restart + visualisation des logs live
  • Dashboard coûts temps réel (par exécution, par projet, par mois)
  • Liste des exécutions récentes avec drill-down
  • État RAG (collections, chunks, dernière ingestion)
  • Alertes sur seuils (budget 80%, circuit breaker, RAG health)

Pas encore commencé. C'est aujourd'hui l'une des priorités post-refonte doc, parce que « script qui tourne en arrière-plan » est une mauvaise UX pour le propriétaire du système, même quand ce propriétaire est aussi son développeur.
